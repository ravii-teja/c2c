# Semantic Layer Strategy

## Why The Semantic Layer Is The Center Of Gravity

In this platform, the semantic layer is not a reporting convenience. It is the mechanism that turns fragmented enterprise assets into something an LLM and a BI workflow can reason about safely.

Without it, the system falls back to brittle source-by-source prompting.

## Design Goals

- unify concepts across inconsistent sources,
- preserve provenance to raw assets,
- support both automated synthesis and human correction,
- encode metrics, dimensions, relationships, and policies,
- improve over time via feedback and drift detection.

## Proposed Semantic Model

The semantic layer should contain at least six object types:

1. Source objects
   Systems, connectors, and access scopes.
2. Asset objects
   Tables, files, document collections, endpoints, and views.
3. Entity objects
   Canonical business nouns such as customer, account, order, ticket, employee.
4. Metric objects
   Canonical business measures with formulas, grain, filters, and provenance.
5. Relationship objects
   Joins, links, equivalence mappings, and evidence-backed associations across assets.
6. Policy objects
   Visibility, sensitivity, retention, and usage constraints.

## Synthesis Pipeline

### Step 1: Normalize Raw Metadata

Convert connector-specific schemas and document metadata into a common representation.

### Step 2: Generate Candidate Concepts

Use heuristics and LLM reasoning to infer:

- candidate entities,
- candidate metrics,
- synonyms,
- likely join keys,
- source overlaps,
- domain labels.

### Step 3: Score And Rank

Every candidate mapping gets:

- confidence score,
- evidence trace,
- source provenance,
- validation status.

### Step 4: Validate

Use deterministic checks wherever possible:

- type compatibility,
- aggregation validity,
- join sanity,
- temporal grain consistency,
- access-policy consistency,
- metric formula checks.

### Step 5: Review And Publish

High-confidence objects can auto-publish to a draft semantic version. Sensitive or high-impact metrics should require human approval.

## Metric Strategy

Metrics should be modeled as contracts, not prompt outputs.

Each metric contract should include:

- business name,
- synonyms,
- definition,
- owner,
- source assets,
- grain,
- formula,
- filter rules,
- time semantics,
- confidence,
- validation history,
- policy annotations.

## Relationship Strategy

Relationship inference is where many AI-over-BI systems fail. We should explicitly support:

- hard joins from keys,
- soft joins from semantic similarity and profile overlap,
- reference links between structured and unstructured assets,
- contradictory relationship flags that require review.

A relationship object should record whether it is:

- verified,
- inferred,
- partial,
- policy-limited,
- stale due to drift.

## Human-In-The-Loop Model

We should assume a review tier for:

- executive metrics,
- compliance-sensitive entities,
- low-confidence joins,
- conflicting metric definitions across teams.

The human loop should refine the system, not replace it. Corrections should feed back into ranking, synonym expansion, and future synthesis runs.

## Versioning

Use semantic-layer versions with:

- draft,
- validated,
- production,
- deprecated.

Each answer generated by the system should record which semantic version it used.

## Success Criteria

The semantic layer is succeeding when:

- new sources can be added without rewriting the whole model,
- the system can answer more questions over time with fewer manual mappings,
- metric disagreements become explicit rather than hidden,
- business users can inspect where a concept came from,
- governance rules remain enforceable across mixed source types.
